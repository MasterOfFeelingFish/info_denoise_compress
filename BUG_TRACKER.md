## P0 - 核心功能阻塞

### BUG-001: 无法添加信息源

**模块**: `bot/handlers/sources.py`

**现象**: 用户通过 `/sources` 尝试添加 Twitter 或网站 RSS 源时失败，无法完成添加操作。

**复现步骤**:
1. 在 Bot 中发送 `/sources`
2. 点击「Twitter」或「网站」按钮
3. 按提示输入一个 Twitter 用户名（如 `@VitalikButerin`）或网站 URL（如 `https://cointelegraph.com`）
4. 观察是否成功添加

**预期结果**: 系统提示添加成功，`/sources` 列表中能看到新增的源。

**实际结果**: 添加失败或无响应。

**验收标准**:
- [ ] 能成功添加一个 Twitter 用户名作为信息源
- [ ] 能成功添加一个网站 RSS URL 作为信息源
- [ ] 添加后 `/sources` 列表中显示新增的源
- [ ] 添加无效 RSS URL 时有明确的错误提示（而不是静默失败）

### BUG-002: 群聊收不到简报 + 群配置夹杂中文

**模块**: `bot/handlers/group.py`, `bot/main.py`

**现象**:
1. 群聊中配置了推送时间，但到时间后群里收不到简报
2. 群聊配置界面中有中文硬编码，非中文用户看到夹杂的中文

**复现步骤**:
1. 把 Bot 拉入一个测试群
2. 在群中发送 `/setup` 配置群聊偏好和推送时间
3. 观察群配置是否正常保存（检查 `data/group_configs/` 目录）
4. 等待配置的推送时间到达，检查群是否收到简报
5. 切换到非中文 Telegram 客户端，查看群配置界面是否有中文

**预期结果**: 群聊按配置时间收到简报；界面文字跟随用户语言设置。

**实际结果**: 群聊收不到简报；配置界面有中文硬编码。

**验收标准**:
- [ ] `/setup` 命令在群聊中可以正常使用
- [ ] 群配置能正确保存到 JSON 文件
- [ ] 推送定时任务能识别群聊并按时发送简报
- [ ] 群配置界面所有文字走 `ui_strings.py` 的多语言系统，无硬编码中文

**相关文件**:
- `bot/handlers/group.py` - 群聊 handler
- `bot/main.py` - 定时推送任务注册
- `bot/locales/ui_strings.py` - 多语言字符串

### BUG-003: 新用户注册后，画像确认消息自动消失

**模块**: `bot/handlers/start.py`

**现象**: 新用户完成 3 轮 AI 对话、生成用户画像后，画像确认消息（显示 AI 总结的用户偏好）刚出现就自动消失了。

**复现步骤**:
1. 用一个新的 Telegram 账号（或清除该用户数据后）向 Bot 发送 `/start`
2. 完成 3 轮 AI 对话（Round 1: 区块链偏好, Round 2: 内容类型, Round 3: 确认画像）
3. 观察 Round 3 结束后的画像确认消息是否稳定显示

**预期结果**: 画像确认消息持续显示，等待用户点击「确认」或「调整」按钮。

**实际结果**: 消息出现后很快自动消失。

**验收标准**:
- [ ] 画像确认消息稳定显示不消失
- [ ] 用户可以正常点击「确认」按钮完成注册
- [ ] 用户可以点击「调整画像」按钮重新调整

**排查方向**: 检查 `confirm_profile` 函数和 `CONFIRM_PROFILE` 状态下是否有 `delete_message` 或 `edit_message_text` 覆盖了之前的消息。也检查是否有并发的 callback 导致消息被替换。

### BUG-004: 两种重复新闻问题

**模块**: `bot/services/content_filter.py`, `bot/services/report_generator.py`

**现象**:
1. **标题和正文重复**: 单条新闻的标题内容和正文摘要内容高度重复（同一段话出现两次）
2. **同一新闻发两遍**: 不同来源的同一条新闻，AI 筛选后没有去重，发送了 2 遍

**复现步骤**:
1. 发送 `/test`（管理员命令）手动触发一次简报生成
2. 查看推送的简报内容
3. 检查是否有标题和摘要内容完全相同的条目
4. 检查是否有不同来源但内容相同的条目

**预期结果**: 每条新闻标题和摘要有区分度；跨源内容已去重。

**实际结果**: 存在标题=摘要的情况；同一事件从不同来源重复出现。

**验收标准**:
- [ ] 单条新闻的标题和摘要不应高度重复（如果 AI 返回的摘要与标题几乎相同，应省略摘要或重新生成）
- [ ] 跨源内容去重有效，同一事件只出现一次（保留信息最丰富的来源）
- [ ] 用 `/test` 触发 3 次以上，每次检查都不应出现明显重复

**相关文件**:
- `bot/services/content_filter.py` - AI 筛选逻辑（两阶段筛选）
- `bot/services/report_generator.py` - 简报生成
- `bot/tests/test_sprint1.py` - 已有去重测试 `TestDeduplication`

## P1 - 用户体验问题

### BUG-005: 等待时间引导文案错误

**模块**: `bot/handlers/start.py`, `bot/locales/ui_strings.py`

**现象**: 注册完成后首次生成简报时，系统提示「预计 10-20 秒」，但实际可能需要 1-3 分钟。用户空等后体验极差。

**相关代码位置**:
- `bot/handlers/start.py` 第 1435 行: `expected_time` 提示
- `bot/locales/ui_strings.py` 第 222 行 (中文) 和第 828 行 (英文): `first_digest_steps`

**修复要求**: 
将提示语改为合理的时间预期，并引导用户不必等待。

**示例修改**:
- 中文: `"首次生成简报可能需要 1-3 分钟，取决于信息源数量。您可以先去忙别的，生成完毕后会自动发送给您。"`
- 英文: `"Generating your first digest may take 1-3 minutes depending on the number of sources. Feel free to do something else - we'll send it to you automatically when it's ready."`

**验收标准**:
- [ ] 所有语言版本的等待提示都已更新为合理时间（1-3 分钟）
- [ ] 提示语引导用户不必等待，可以先离开
- [ ] 检查代码中所有出现 `10-20` 秒的地方都已修正

### BUG-006: 小语种 UI 未翻译

**模块**: `bot/locales/ui_strings.py`

**现象**: 中文和英文以外的语言（日语、韩语等），部分 UI 界面文字没有对应翻译，显示英文 fallback 或中文。

**复现步骤**:
1. 在 Telegram 设置中将语言改为日语或韩语
2. 与 Bot 交互，逐一检查各页面
3. 记录哪些地方显示了非目标语言的文字

**验收标准**:
- [ ] `ui_strings.py` 中已支持的语言（zh/en/ja/ko）覆盖所有 key
- [ ] 新增的群聊、付费相关 UI key 都有对应翻译
- [ ] 不存在硬编码的中文字符串（除了 prompt 模板）

### BUG-007: 输入 "meme" 等英文关键词时默认英语界面

**模块**: `bot/handlers/start.py`

**现象**: 新用户 onboarding 第一轮时，如果输入英文关键词（如 "meme"），系统可能将用户语言设置为英语。

**修复建议**: 注册开始时就让用户选择语言，或仅根据 Telegram 客户端语言设置来确定，不根据用户输入内容推断语言。

**验收标准**:
- [ ] 用户语言设置不受 onboarding 对话内容影响
- [ ] 语言仅由 Telegram 客户端设置或用户手动选择决定

## P2 - 视觉呈现问题

### BUG-008: 简报视觉呈现优化

**模块**: `bot/services/report_generator.py`

**现象** (多个子问题):
1. **破折号占两行**: 中文破折号 `——` 或 `—` 在 Telegram 中可能折行显示不美观
2. **标题原文链接和「查看原文」重复**: 标题本身已经是原文链接，底部又有「查看原文」按钮，重复展示
3. **噪音前缀未清理**: 显示 "blockbeats消息"、"转发xxx"、"深潮 TechFlow 消息" 等来源前缀
4. **第二部分（原文列表）占 3/4 篇幅**: AI 摘要应该是重点，但视觉上被原文列表淹没

**验收标准**:
- [ ] 破折号统一处理，不出现占两行的情况
- [ ] 标题链接和「查看原文」不重复展示（二选一）
- [ ] 清理常见噪音前缀（blockbeats、TechFlow、转发等）
- [ ] AI 摘要部分在视觉上更突出（分段落、分主题）
- [ ] 原文列表精简展示，不喧宾夺主

**相关代码**: `report_generator.py` 第 722-724 行已有部分破折号处理逻辑。

## P3 - 功能需求（付费体系）

### BUG-009: 付费功能体系完善

**模块**: `bot/handlers/payment.py`, `bot/utils/permissions.py`, `bot/config.py`

**需求描述**: 建立基于环境变量控制的付费功能开关体系，不在代码中硬编码付费/免费的区别。

**功能点**:
1. **通过 TG Bot 自带的付费入口收费** — `payment.py` 已有框架，需要测通
2. **收费用户可自主提交关注的信息源** — 用户输入大概描述即可（不要求标准格式匹配），后台自动转换
3. **收费用户可绑定推特，导入关注列表** — 新功能
4. **免费用户只看统一配置的信息源** — 通过权限检查控制
5. **免费用户推荐新用户可抵扣一次付费功能** — referral 机制

**技术要求**:
- 所有付费/免费功能差异通过环境变量 + `permissions.py` 控制
- 新增环境变量示例:
  ```
  FEATURE_PAYMENT=true
  FEATURE_CUSTOM_SOURCES=premium    # free|premium|disabled
  FEATURE_TWITTER_IMPORT=premium
  FEATURE_REFERRAL=true
  ```
- 后台可通过修改用户 JSON 数据中的 `plan` 字段来标记付费用户

**验收标准**:
- [ ] TG Bot 付费入口可以正常发起支付流程
- [ ] 付费用户可以用自然语言描述想关注的信息源（不必精确输入 RSS URL）
- [ ] 免费用户尝试使用付费功能时，提示升级
- [ ] 所有功能开关可通过 `.env` 环境变量控制，无硬编码
